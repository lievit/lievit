# Documentation plan: which docs exist and why

This file is the **plan** for lievit's documentation and living-docs discipline. It is wired up
when the Maven build lands; right now it states the intent so the structure is decided before the
code arrives. The discipline mirrors the sister repos `spring-aiact` and `spring-gdpr`.

## Principle: the code is the source of truth

The living documentation is **generated from the code, never hand-edited**. To change a generated
doc, you change the code (add a `@spec` javadoc, rename a test, name a constant). This is the same
contract the sister repos and the gest client repo run under: generated docs cannot drift from the
code because they are derived from it.

## Living docs (generated by tracegate, when the build lands)

lievit will use [**tracegate**](https://github.com/iambilotta/tracegate), pinned to a commit, the
same generator the sister repos and the gest client repo consume. It parses Java / TS via
tree-sitter (real AST) and emits a per-module `_generated/` catalog:

- `requirements.md` — the FR / NFR / INV / CON catalog, derived from `@spec.*` javadoc on `@Test`
  methods.
- `requirements-by-us.md` — a User Story coverage view.
- the as-built set (endpoints, events, modules, schema, config, ports, templates, coverage, todo,
  adr-index, dependencies) as the runtime modules grow.

A `tracegate.toml` at the repo root will register the test-bearing modules explicitly (so the
committed catalog is deterministic across machines and CI). A `make requirements` target
regenerates it; `make requirements-check` is the CI drift-gate (exit 2 on drift). All of this is
**deferred until the Maven build exists**; this section is the target, not the current wiring.

## TDD discipline (when code starts)

- **Spec first**: a frozen spec becomes a failing test (RED).
- **Minimal code to green** (GREEN), then refactor while green (REFACTOR).
- **Traceability**: every `@Test` carries a structured `@spec.given` / `@spec.when` / `@spec.then`
  javadoc; the requirement ID is derived from package + class + method.

## Test taxonomy (load-bearing for the gates)

The category is derived from the test class-name suffix (no tags to maintain), the same convention
as the gest client repo:

| Class suffix | Category | Meaning |
|---|---|---|
| `*Test.java` | **FR** | Functional requirement (default). |
| `*NfrTest.java` / `*NfrIT.java` | **NFR** | Non-functional (perf, security, observability, resilience). |
| `*InvariantTest.java` / `*InvariantIT.java` | **INV** | Domain / protocol invariant (wire codec, port contracts). |
| `*ContractTest.java` / `*ContractIT.java` | **CON** | HTTP boundary contract (the wire endpoint). |

The `*IT.java` suffix is orthogonal: it means "needs Spring + a real boot", not a category.

The load-bearing invariant tests for lievit's core: the wire snapshot **roundtrips** (encode then
decode yields the same state), is **tamper-evident** (a modified snapshot fails HMAC verification),
and is **replay-safe** (an expired snapshot is rejected). These are the golden roundtrip triples
described in ADR-0007.
